# Phase 2F-E2C Production Schedule move UI

The private `/production-schedule` route continues to render the shared Production Board model. `production=view` receives the normal read-only shared view; only `production=use` receives the explicit reschedule interaction contract. The public `/production-board` route remains unauthenticated and read-only and imports no mutation action.

Both Board routes use eight fixed calendar workweeks with Monday through Friday as the five columns and the current Vancouver workweek as the default first row. The private Schedule provides Previous week, Today, Next week, and Go to date controls backed by a normalized `?week=YYYY-MM-DD` Monday anchor. Past weekdays remain visible with a darker day treatment, today stays highlighted in its actual weekday position, and future weekdays retain the normal treatment. Completed bookings remain visible, readable, labelled **Completed**, muted, and immovable; past unfinished bookings remain prominent and movable when the user has `production=use`. Saturday and Sunday bookings stay in their week’s separate weekend-exception area.

On a desktop-capable fine pointer, the whole eligible booking card can be dragged to a visible date column. The highlighted destination keeps its closure and overload presentation, and a valid drop lands the card locally before review. Dragging outside a date, dropping on the source date, cancelling, or receiving an unconfirmed server result leaves or returns the card to its authoritative source position. The accessible **Move** action uses the same state machine and a native date input for phone, keyboard, and outside-window destinations; touch dragging is not required.

One modal review dialog presents only the applicable requirements and warnings. A source on or before Vancouver today requires **The whole job was not started.** A destination before today requires a trimmed 1–500 character reason. A closed destination requires the explicit closed-date override. Projected overload and unknown capacity are visible but nonblocking. An authenticated, permission-scoped, read-only preview derives booking hours on the server and reuses normalized capacity reads for destinations outside the rendered window. The E2B RPC remains independently authoritative.

Closed-date override authority comes only from `dg_daily_capacity.is_closed`; display closure and scheduling closure are intentionally separate concepts. Browser move guidance mirrors the E2B booking-ID and Shop Hours boundaries, while the RPC remains authoritative. Outside-window attempts keep the originating card visibly pending until the attempt finishes or is abandoned.

Each attempt receives one secure command UUID. An unchanged retry after an unavailable response retains it; a material request change after failure rotates it. Success, cancellation, stale state, or abandonment discards it. Stale state reverts local placement, refreshes the Board, and reports that the booking changed elsewhere. Success keeps the destination placement while authoritative data refreshes and uses the reusable top-center confirmation toast, whose success message remains for about one second.

Completed and locked bookings are not draggable and have disabled Move controls with an explanation. E2C adds no Started status, placeholders, booking creation, Shop Hours editing, completion controls, job changes, scheduling notes, Calendar synchronization, checkpoint mutation, or capacity mutation.

This workflow is available for controlled Vercel parallel testing only. The current Apps Script DoorGo and Google Calendar remain the live operational system until every replacement workflow is complete and final cutover is approved.
